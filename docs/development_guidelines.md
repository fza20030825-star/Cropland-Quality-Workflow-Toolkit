# Cropland Quality Workflow Toolkit 开发规范指南

本文档面向后续维护者和 AI agent。修改代码前先阅读本文件，并以当前代码和 `docs/workflow_manual.qmd` 为准；如果实现发生变化，必须同步更新用户文档和本指南中相关条目。

## 总体原则

- 只保留统一 UI 作为正式入口：`Start_CroplandQualityWorkflow.cmd` 和 `run_workflow_ui.py`。不要恢复旧版单步 UI 入口。
- 正式业务流程固定为四步：计算高标隶属度、更新隶属度、更新三调图斑、面积平差。
- 辅助工具可以存在于统一 UI 中，但不得改变四步主流程顺序。
- 代码改动必须保持输入数据只读，任何结果都单独输出。
- 所有运行过程文件必须在成功或失败后清理。
- 每次改变算法、字段、交互、审查、日志或输出行为，都要同步修改 README 和 `docs/workflow_manual.qmd`。

## 统一 UI 规则

- 顶部流程选择只负责切换四个正式流程和辅助工具。
- 底部区域统一命名为“详细信息”，所有任务状态、日志路径、失败原因和自动填写信息都写入这里。
- 各页面不设置独立的完整审查按钮。用户点击“提交...”后自动执行提交前审查。
- 审查失败时只弹出审查报告并停止提交，不再叠加额外错误弹窗。
- 审查通过时只弹出一次确认报告，用户确认后才提交后台任务。
- 正式流程完成后自动填写下一步输入并跳转到下一步。
- 辅助工具完成后可以自动填写后续输入，但不得自动跳转离开当前辅助页面。
- “恢复默认输入”只恢复输入路径、图层选择、投影选择、输出名称和参数，不清空详细信息。
- 统一输出 GDB 要全局同步：顶部选择一次后写入所有步骤；任意步骤内修改输出 GDB，也同步回其他步骤。
- 国标二级农业区要全局同步：任意带农业区选择的页面改选后，其他相关页面自动改为同一农业区，并清除旧审查报告。
- 表格、文本框、列表等可滚动子控件必须有可见滚动条；鼠标位于这些子控件上时，滚轮不得同时滚动外层页面。

## 输出和字段

- 强烈建议并优先使用 FileGDB。涉及完整中文字段名、固定字段顺序或新增字段的流程不得依赖 Shapefile。
- 允许输出到输入所在同一 GDB，但输出要素类名必须不同；禁止覆盖任何输入数据。
- 第二步和第三步输出必须是固定 39 字段，字段名和顺序与手册一致。
- 第一步输出为 13 个指标原值、13 个 `F...` 字段、`评价得分`、`质量等级`；标记重叠模式额外输出 `是否重叠`。
- 第四步新增 `等级面积` 字段，别名为 `等级*面积`，并写入平差面积和县域质量等级结果。
- 老数据字段名可能被截断，匹配字段时应使用宽松但可解释的开头包含原则；如果一个短字段可能匹配多个指标，必须判为歧义并打回。
- `pH`、`PH`、`ph`、`Ph` 都视为 `酸碱度`。
- 非海拔必需农业区允许 `海拔高度` 和 `F海拔高度` 为空；秦岭大巴山林农区、渝鄂湘黔边境山地林农牧区必须检查海拔完整性。

## 空间匹配和确定性

- 坐标系不一致时，用户必须选择统一投影来源；只投影过程文件，不改输入。
- 第一、二、三步都要在提交前显示或确认投影。第一步勾选数据后应立即刷新投影表，不依赖提交前审查。
- 所有空间赋值逻辑必须尽量百分百可复现，避免隐式随机。
- 第三步更新三调图斑时：只要有任何重叠，就按最大重叠面积赋值；完全无重叠时按最近距离赋值；距离相同按上一步结果面积更大者赋值；仍完全相同才允许随机，并必须在详细信息和日志中警告。
- 第一步跨来源完全相同几何和 13 个指标的重复要素直接去重。
- 第一步非完全相同重叠有两种模式：标记重叠并输出 `是否重叠`；或按输入顺序 Erase，前序数据源优先保留。
- 标记重叠模式完成后必须统计最终输出中 `是否重叠=1` 的要素数量，并写入详细信息、日志和历史记录。

## 审查和报错

- 提交前审查是所有工具的硬门槛，不能为了简化 UI 绕过。
- 审查报告必须指出文件、字段、要素和问题类型；问题过多时可以汇总并列样例。
- 输出完成后还要审计必须计算或更新的字段，发现空值或未更新要打回。
- 打回报告应尽量使用要素层面的 OID 和前 10 个字段值，便于专业人员定位。
- 审查失败不提交后台任务；审查报告是主要反馈，不要再弹重复错误窗。
- 输出安全、缺少运行环境、读文件失败、打开日志失败等非审查类错误仍可用错误弹窗。

## 日志和历史

- 每个工具都要写日志和 JSONL 历史记录。
- 日志要包含输入、输出、规则文件、投影来源、审查报告、统计结果、错误堆栈和过程文件清理信息。
- 详细信息区要展示用户最需要立即知道的结果，例如输出路径、日志路径、平差系数、最终质量等级、重叠数量。
- 日志、历史记录、过程文件和缓存不纳入 Git 管理。

## 启动和环境

- 普通用户通过 `Start_CroplandQualityWorkflow.cmd` 双击启动。
- 启动器必须查找可导入 `arcpy` 的 ArcGIS Pro Python 环境，并在进入 UI 前让用户确认。
- 缓存的 Python 路径只用于加速查找，每次启动仍要询问是否使用当前环境。
- 不要把开发者个人复制出的 conda 环境路径写死进代码或文档。

## 开发和验证

- 手工编辑文件使用 `apply_patch`。
- 不要删除或回退用户未要求处理的改动。
- 对 UI 行为、字段、审查、空间规则的改动，至少运行：

```powershell
& 'C:\Program Files\ArcGIS\Pro\bin\Python\envs\arcgispro-py3\python.exe' -m compileall -q src run_workflow_ui.py
$env:PYTHONPATH='src'; & 'C:\Program Files\ArcGIS\Pro\bin\Python\envs\arcgispro-py3\python.exe' -m pytest
git diff --check
```

- 测试或编译生成的 `__pycache__`、`.pytest_cache` 要清理。
- 如果新增工具或规则，必须补充用户文档、历史记录、日志路径、输出安全检查和必要测试。
